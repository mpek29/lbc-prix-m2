# Publishing

Three ways to get this into a browser, in increasing order of effort and
permanence.

| Route                   | Permanent | Public listing | Cost      | Wait           |
| ----------------------- | --------- | -------------- | --------- | -------------- |
| Temporary add-on        | No        | No             | Free      | None           |
| Self-distribution (AMO) | Yes       | No             | Free      | About a minute |
| Store listing           | Yes       | Yes            | See below | Hours to days  |

## 1. Temporary add-on (Firefox, for testing)

`about:debugging#/runtime/this-firefox` then **Load Temporary Add-on**, and pick
either `.output/firefox-mv3/manifest.json` or a built `.zip`. Gone at restart.

Chrome equivalent: `chrome://extensions`, Developer mode, **Load unpacked**,
pick `.output/chrome-mv3`. That one survives restarts but shows a warning
banner on every launch.

## 2. Self-distribution signing (Firefox, permanent, no listing)

Release-channel Firefox refuses unsigned extensions, and the
`xpinstall.signatures.required` pref does nothing there. It only works on
Developer Edition, Nightly and ESR. So a permanent install on a normal Firefox
needs a signature, even for personal use.

Mozilla signs unlisted add-ons for self-distribution, free, usually in under a
minute, without any public page.

1. Get API credentials at
   <https://addons.mozilla.org/en-US/developers/addon/api/key/> (needs a Firefox
   Account).
2. Export them. Never commit them:

   ```bash
   export WEB_EXT_API_KEY='user:12345678:123'
   export WEB_EXT_API_SECRET='...'
   ```

3. Sign:

   ```bash
   npm run sign:firefox
   ```

The signed `.xpi` lands in `.output/signed/`. Open it with Firefox, or drag it
onto the window, and it installs for good.

The extension id in `wxt.config.ts` (`lbc-prix-m2@mpek29.github.io`) is what ties
a signature to this add-on. Changing it after publishing orphans every existing
install, so treat it as permanent.

## 3. Store listings

What most people mean by "released": a public page anyone can install from.

### Before either store

Both want the same set of things, and neither is generated by the build:

- **Screenshots** of the extension doing its job. Take them on a real leboncoin
  search page, not on the harness. The harness is for development and looks
  synthetic.
- **A short summary** and a longer description.
- **A privacy statement.** Ours is short: no data collected, no network
  requests, one boolean in local storage. The Firefox manifest already declares
  `data_collection_permissions: { required: ['none'] }`.
- **A justification for every permission.** We request exactly one, `storage`,
  used to remember whether badges are switched on.

### addons.mozilla.org

Free. Needs a Firefox Account.

1. <https://addons.mozilla.org/developers/addon/submit/>, choose **On this
   site** for a listed add-on.
2. Upload `lbc-prix-m2-<version>-firefox.zip` from the GitHub release.
3. **Upload the sources archive too.** This is not optional for us. AMO requires
   source code whenever the submitted files are generated by a build step, and
   ours are bundled by Vite. `npm run zip:firefox` produces
   `lbc-prix-m2-<version>-sources.zip` for exactly this. Reviewers also need
   build instructions: Node version from `.nvmrc`, then `npm ci` and
   `npm run build:firefox`.
4. Fill in the listing, submit, wait. Automated validation runs first. A human
   review may follow.

### Chrome Web Store

**There is a one-time 5 USD developer registration fee.** It is per account, not
per extension, and it is the thing that surprises people.

1. Register at <https://chrome.google.com/webstore/devconsole>.
2. Upload `lbc-prix-m2-<version>-chrome.zip` from the GitHub release.
3. Provide the listing assets. Sizes are strict:
   - Screenshots: 1280x800 or 640x400
   - Small promo tile: 440x280
4. Complete the privacy practices form, including a justification for the
   `storage` permission and a declaration that no user data is collected.
5. Submit. Review usually takes a few days.

### After a listing exists

Version numbers cannot be reused on either store, so a rejected or withdrawn
upload still burns its number. Bump `package.json`, update `CHANGELOG.md`, tag,
and let the release workflow rebuild rather than reusing an old artefact.
